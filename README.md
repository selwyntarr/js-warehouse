# Sales Data Warehouse

A star-schema data warehouse built from raw sales CSVs, with a Python ETL pipeline and SQL analytical queries.

## Requirements

- Python 3.8+
- pandas (`pip install -r requirements.txt`)

## Setup & Run

```bash
# 1. Create and activate a virtual environment
python3 -m venv .venv
source .venv/bin/activate      # Windows: .venv\Scripts\activate

# 2. Install dependencies
pip install -r requirements.txt

# 3. Run the ETL pipeline
python pipeline.py
# if the above doesn't work, try:
python3 pipeline.py
```

The pipeline reads from `data/`, creates `warehouse.db` in the project root, and prints a row-count summary on completion. Running it multiple times is safe — it will not duplicate data.

## Verify the Database

```bash
# Open the SQLite shell
sqlite3 warehouse.db

# Check tables
.tables

# Spot-check row counts
SELECT COUNT(*) FROM fact_orders;
SELECT COUNT(*) FROM dim_customer;
SELECT COUNT(*) FROM dim_product;

# Exit
.quit
```

## Run the SQL Queries

```bash
sqlite3 warehouse.db < queries.sql
```

Or paste individual queries from `queries.sql` into the SQLite shell.

## Assumptions

- `order_date` is stored as-is on the fact table; month/year grouping is done via `strftime()` in SQL. A full `dim_date` table was deliberately omitted per the scope note in the spec.
- Order 1015 has `quantity = 0` — this row is kept in the database with `revenue = 0` and `profit = 0`, and a warning is emitted during the pipeline run. It is not silently dropped.
- `cost_price` is treated as a product attribute and stored on `dim_product`. Profit is computed at ETL time and stored on `fact_orders` for query performance.
- The `is_returned` flag is derived from `status = 'returned'` and stored on the fact table as a convenience for filtering without repeated string comparisons.

## What I Would Add With More Time

- **`dim_date` table** — year, month, quarter, day-of-week columns for richer time-based slicing without `strftime()` workarounds.
- **SCD Type 2 for dimensions** — track customer segment or product category changes over time with effective-date columns.
- **Incremental loads** — currently a full reload; a real pipeline would process only new/changed rows using a high-watermark on `order_date`.
- **Data quality report** — write flagged rows to a `rejected_rows` table instead of just printing warnings, so bad data is auditable.
- **Tests** — pytest unit tests for the transformation functions (revenue, profit, is_returned) and an integration test that loads a fixture CSV and asserts the DB state.
